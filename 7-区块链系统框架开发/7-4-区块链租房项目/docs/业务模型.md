# 链信租房（HKZF）业务模型

本文档描述项目的**角色分工、租赁办理流程、链上数据与接口映射**，作为开发与演示的 stable 参考。

## 1. 项目定位

基于 Hyperledger Fabric 的**区块链租房可信办理平台**（课程/演示级原型）：

- **管理端**：维护用户无法自证的**权威数据**（身份、房产），写入认证链码
- **用户端**：发起**租赁申请**，按序完成身份核验 → 房产核验 → 合同存证
- **后端**：在用户签约时**重新查链复核**前两步，通过后才允许合同哈希上链

链上仅存认证结果与合同 SHA256 摘要，不存 CSV 原文、不存 PDF 全文。

---

## 2. 角色分工

| 角色 | 入口 | 职责 | 数据操作 |
|------|------|------|----------|
| 管理员 | `admin.html` | 导入公安/房管权威数据 | **写** `authentication`、`certification` 链码 |
| 租客/用户 | `index.html` 办理中心 | 发起并完成租赁申请 | **查** 认证链码；**写** `contract` 链码（经后端校验） |

原则：**管理端管「信什么数据」；用户端管「办什么事」。** 合同由用户签约产生，不由管理端日常上传。

---

## 3. 业务主流程

```
┌─────────────────────────────────────────────────────────────┐
│  管理端（前置准备）                                            │
│  admin.html → 上传 auth.csv → 上传 house.csv（各约 5 秒上链）   │
└────────────────────────────┬────────────────────────────────┘
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  用户端 · 租赁申请（自动生成编号 RZ…）                          │
│                                                             │
│  ① 身份认证  auth.html                                      │
│     输入姓名 + 身份证 → GET /api/auth → 查 authentication    │
│                             │                               │
│  ② 房产认证  house.html   （须 ① 通过）                      │
│     输入房产证 + 身份证 → GET /api/house → 查 certification  │
│                             │                               │
│  ③ 签约存证  contract.html （须 ①② 通过）                    │
│     上传合同文件 → POST /api/contract                        │
│     后端复核身份+房产 → 写 contract 链码（SHA256）            │
│                             │                               │
│  完成 → index.html 查看申请状态「租赁办理完成」                 │
└─────────────────────────────────────────────────────────────┘
```

### 3.1 步骤门禁

| 规则 | 实现位置 |
|------|----------|
| 未通过身份认证，不可进入房产页 | 前端 `HKZF.guardStep(2)` |
| 未通过房产认证，不可进入签约页 | 前端 `HKZF.guardStep(3)` |
| 步骤条中未解锁步骤不可点击 | 前端 `HKZF.canAccessStep()` |
| 合同上链前须链上身份、房产均有效 | 后端 `verifyAuthOnChain` / `verifyHouseOnChain` |

申请进度保存在浏览器 `sessionStorage`（键 `hkzf_flow`），字段包括：

`rentalId`、`name`、`id`、`houseId`、`contractId`、`contractHash`、`authDone`、`houseDone`、`contractDone`

## 4. 链码与数据模型

### 4.1 authentication（个人认证）

| 项目 | 说明 |
|------|------|
| 管理端写入 | CSV：`身份证号, 姓名, 是否有不良记录` |
| 链上 Key | 身份证号 |
| 链上 Value | `姓名:是否有不良记录`（如 `张三:false`） |
| 用户查询 | `check(姓名, 身份证号)` → `true:false` |
| 通过条件 | 身份匹配 **且** 无不良记录（`true:false`） |

### 4.2 certification（房产认证）

| 项目 | 说明 |
|------|------|
| 管理端写入 | CSV：`房产证号, 身份证号, 是否可出租` |
| 链上 Key | 房产证号 |
| 链上 Value | `身份证号:是否可出租` |
| 用户查询 | `check(房产证号, 身份证号)` → `true:true` |
| 通过条件 | 产权人匹配 **且** 可出租 |

### 4.3 contract（合同存证）

| 项目 | 说明 |
|------|------|
| 写入方 | 仅用户端（经后端校验后） |
| 链上 Key | 合同编号（上传文件名去掉扩展名，如 `HT2026001`） |
| 链上 Value | 合同文件 SHA256 十六进制字符串 |
| 查询 | `get(合同编号)` → 哈希值 |

三步数据在链上**相互独立**；业务关联由**租赁申请单 + 后端复核**保证，而非单一链码记录。

## 5. HTTP 接口

| 方法 | 路径 | 角色 | 说明 |
|------|------|------|------|
| POST | `/auth` | 管理端 | 上传 `auth.csv`，延迟任务写链 |
| GET | `/auth` | 用户端 | 参数 `name`、`id`，身份核验 |
| POST | `/house` | 管理端 | 上传 `house.csv`，延迟任务写链 |
| GET | `/house` | 用户端 | 参数 `houseId`、`id`，房产核验 |
| POST | `/contract` | 用户端 | 表单：`contract` 文件 + `name` + `id` + `houseId`；复核通过后上链 |
| GET | `/contract` | 用户端 | 参数 `contractId`，查询链上哈希 |

Web 经 Nginx 反代为 `/api/*`。

## 6. 演示路径（样本数据）

样本文件见 `hkzf-front/samples/`。

1. **管理端**：导入 `auth.csv` → 等待约 5 秒 → 导入 `house.csv` → 再等待约 5 秒  
2. **用户端首页**：点击「开始身份认证」，生成租赁申请编号  
3. **身份认证**：张三 / `211004197001010000` → 预期 `true:false`  
4. **房产认证**：房产证 `1111112222223333333000000000`，身份证同上 → 预期 `true:true`  
5. **签约存证**：上传 `HT2026001.pdf` → 合同编号 `HT2026001`，返回链上哈希  

### 反面用例

- 王五（`211004199003030000`）：有不良记录，身份认证不通过  
- 跳过前两步直接打开 `contract.html`：重定向回前置步骤  
- 不带 `name/id/houseId` 调用合同上传：HTTP 403

## 7. 部署与访问

```bash
./deploy.sh up          # 全栈（minikube + 链 + 业务 + port-forward）
./deploy.sh build-app   # 仅构建 API / Web 镜像
./deploy.sh forward     # 单独启动 Web 转发
```

默认访问：`http://127.0.0.1:30888/`

---

## 8. 已知边界与扩展方向

当前为**可信办理原型**，尚未包含：

- 用户登录与权限隔离  
- 租赁单号写入链码（三步打包为一条链上记录）  
- 电子签章、租金/租期/房源发布等业务模块  
- 合同原文链下持久化与哈希比对 UI  

扩展时建议保持：**权威数据管理端写入，用户动作用户端触发，关键写链操作后端统一校验。**
